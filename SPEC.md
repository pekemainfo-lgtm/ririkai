# 1. プロダクト概要
## 1.1 プロダクト名
ReRIKAI
日本語表記：リリカイ
## 1.2 プロダクトの目的
ReRIKAIは、学習者が教材を読んだり問題を解いたりした後、自分の言葉で内容を説明し、その説明をAIが分析する学習支援サービスである。
AIが教材の代わりに学習するのではない。
学習者本人が説明し、AIは以下を支援する。
* 何を理解できているか
* 何が曖昧か
* どこに誤解の可能性があるか
* 何を復習すべきか
* 前回から何が改善したか
中心思想は次のとおり。
> 「勉強したか」ではなく、「自分の言葉で説明できるか」を確認する。
## 1.3 初期対象
初期版では、主に資格試験学習を対象とする。
例：
* AWS認定
* LinuC
* CCNA
* 社会保険労務士
* 行政書士
* 簿記
* その他、言語化による理解確認が有効な学習
数学の計算問題や数式入力への本格対応は、初期版の対象外とする。

# 2. 今回の開発方針
## 2.1 開発環境
初期版はWindows上で開発する。
想定環境：
* Windows
* Claude Code
* VS Code
* Git
* Node.js
* AWS CLI
Mac mini、Xcode、Swift、Apple Speechは、今回の実装には使用しない。
## 2.2 初期版の文字起こし方式
初期版では、アプリ内録音および自動文字起こしを実装しない。
利用者は、iPhoneなどで録音・文字起こしした文章を、ReRIKAIの入力欄へ貼り付ける。
初期フロー：
```text
iPhoneで録音
↓
iPhone側で文字起こし
↓
文字起こし結果をコピー
↓
ReRIKAIのWeb画面へ貼り付け
↓
AI分析
↓
Markdown生成
↓
保存
↓
カレンダーへ反映
↓
復習カード候補を生成
```
## 2.3 将来の文字起こし方式
将来的には次の構成を追加する。
```text
WebまたはiPhoneから音声をアップロード
↓
AWSに一時保存
↓
Mac miniがジョブを取得
↓
Apple Speechで文字起こし
↓
文字起こし結果をAWSへ返却
↓
既存のAI分析・Markdown生成処理へ渡す
```
そのため、今回の実装では、文字起こしの入力元と学習処理を分離する。
以下のどちらでも、共通の`transcript`文字列を受け取れること。
```text
手動貼り付け ─────┐
                  ├→ 共通学習セッション処理
Apple Speech ─────┘
```
AI分析、Markdown生成、カード生成、保存処理を、手動入力専用の実装にしないこと。

# 3. システム構成
## 3.1 初期構成
```text
利用者のブラウザ
↓
ReRIKAI Webアプリ
↓
AWS API
├─ AI分析処理
├─ Markdown生成処理
├─ カード生成処理
├─ 重複カード判定
└─ 復習スケジュール処理
↓
S3
├─ 学習セッションMarkdown
└─ ノート写真
↓
DynamoDB
├─ 学習セッション索引
├─ カード
└─ 復習履歴
```
## 3.2 重要な設計方針
* 学習内容の正本はMarkdownとする
* MarkdownはS3に保存する
* DynamoDBには検索・一覧・カレンダー表示用の索引を保存する
* 長文の文字起こし本文をDynamoDBへ保存しない
* AIが学習者の発言を正しい内容に書き換えない
* 過去の学習Markdownは、後からカードの正解が変わっても書き換えない
* カードは現在の復習用知識として更新可能にする

# 4. データの正本
## 4.1 Markdownを正本とする
1回の学習セッションにつき、1つのMarkdownファイルを作成する。
保存例：
```text
sessions/users/{userId}/2026/07/18/session_{sessionId}.md
```
S3オブジェクトのContent-Type：
```text
text/markdown; charset=utf-8
```
S3 Versioningを有効化できる構成であれば使用する。
ただし、既存バケットに対する設定変更が必要な場合は、影響を確認してから行うこと。
## 4.2 Markdownに保存する情報
Markdownには最低限、次の情報を保存する。
* セッションID
* ユーザーID
* 学習日
* 開始時刻または作成時刻
* 資格・教材分類
* 分野
* テーマ
* 学習時間
* 今日の目的
* 学習者が分からなかった点
* 生の文字起こし
* 表記のみ整えた文字起こし
* AI分析
* 理解できている点
* 曖昧な点
* 誤解の可能性
* AIからの確認質問
* 復習すべき事項
* カード候補
* 採用されたカードID
* ノート写真参照
* 作成日時
* 更新日時
* Markdownスキーマバージョン
## 4.3 Front Matter例
```yaml
---
schemaVersion: 1
sessionId: session_01JXYZ
userId: user_123
studyDate: 2026-07-18
qualification: AWS SOA-C03
subject: Monitoring
topic: CloudWatch and CloudTrail
durationMinutes: 25
understandingScore: 68
status: completed
createdAt: 2026-07-18T14:30:00+09:00
updatedAt: 2026-07-18T14:30:00+09:00
cardIds:
  - card_01JAAA
  - card_01JBBB
noteImages:
  - notes/users/user_123/2026/07/18/session_01JXYZ/note_01.jpg
---
```
## 4.4 Markdown本文例
```markdown
# CloudWatchとCloudTrail
## 今日の目的
CloudWatchとCloudTrailの違いを自分の言葉で説明する。
## 自分で分からなかった点
- CloudTrailがどこまで監視するのか曖昧
- CloudWatch Logsとの関係が曖昧
## 生の文字起こし
ここには利用者が貼り付けた文章を、そのまま保存する。
## 表記を整えた文字起こし
IAM、EC2、CloudWatchなど、明白な表記揺れのみ修正した文章を保存する。
## AI分析
### 理解できている点
- CloudWatchがメトリクスやログを扱うことは説明できている。
### 曖昧な点
- CloudTrailとCloudWatch Logsの役割が混ざっている可能性がある。
### 誤解の可能性
- CloudTrailがCPU使用率を監視すると捉えている可能性がある。
### 確認質問
1. EC2のCPU使用率を確認するサービスは何か。
2. AWS APIの操作履歴を確認するサービスは何か。
## 復習事項
- CloudWatch
- CloudTrail
- CloudWatch Logs
- メトリクスとAPI履歴の違い
## 関連カード
- card_01JAAA
- card_01JBBB
```

# 5. AIによる文字起こし処理ルール
## 5.1 原文を保存する
利用者が貼り付けた原文は、必ず`rawTranscript`として保存する。
AIが原文を上書きしてはならない。
## 5.2 表記修正は限定する
AIは、明白な固有名詞や用語の表記修正のみ行ってよい。
許可例：
```text
アイアム
↓
IAM
```
```text
イーシーツー
↓
EC2
```
禁止例：
```text
CloudTrailはCPU使用率を監視する
↓
CloudWatchはCPU使用率を監視する
```
後者は意味の修正であり、学習者の理解状態を消してしまうため禁止する。
正しくは、原文を残したうえでAI分析欄に次のように記載する。
```text
CloudTrailとCloudWatchを混同している可能性があります。
```
## 5.3 AIの役割
AIは以下を行う。
* 理解できている点の抽出
* 曖昧な点の抽出
* 誤解の可能性の指摘
* 不足している説明の指摘
* 確認質問の生成
* 復習事項の抽出
* カード候補の生成
* 前回セッションとの比較
AIは以下を行わない。
* 学習者の説明を正解へ書き換える
* 不明な内容を勝手に補完する
* 根拠なく高得点を付ける
* 文字数の多さだけで理解度を判断する
* カードを無制限に大量生成する

# 6. 学習セッション入力画面
## 6.1 入力項目
最低限、以下を入力できること。
* 学習日
* 資格・コース
* 分野
* テーマ
* 学習時間
* 今日の目的
* 文字起こし貼り付け欄
* 自分で分からなかった点
* ノート写真
* カードを生成するかどうか
## 6.2 入力画面の方針
* スマートフォンでも入力しやすいこと
* 文字起こし欄は長文入力に耐えられること
* 自動保存または入力消失対策を検討すること
* 保存中の二重送信を防止すること
* エラー時に入力内容を失わないこと
* 必須項目と任意項目を明示すること
## 6.3 保存処理
保存ボタン押下後、次を実行する。
1. 入力値検証
2. 学習セッションID生成
3. AI分析
4. Markdown生成
5. S3保存
6. DynamoDB索引登録
7. カード候補生成
8. 重複カード候補検索
9. カード確認画面表示
10. カレンダーへ反映
AI分析またはカード生成に失敗した場合でも、入力した学習記録自体を失わないこと。
可能であれば、次のように段階的な状態を持つ。
```text
draft
processing
analysis_completed
cards_pending
completed
failed
```

# 8. ノート写真
## 8.1 初期版の扱い
初期版ではノート写真をS3に保存する。
以下は実装しない。
* OCR
* 手書き文字認識
* AIによる内容分析
* 自動要約
* 自動カード化
目的は、後から学習内容を見返すための参照資料とすることである。
## 8.2 アップロード
可能であれば、Presigned URLを利用してブラウザからS3へ直接アップロードする。
LambdaやAPIサーバーを画像本体の中継に使用しない。
## 8.3 保存例
```text
notes/users/{userId}/2026/07/18/{sessionId}/note_{imageId}.jpg
```
## 8.4 制限
最低限、以下を考慮する。
* 許可する画像形式
* 最大ファイルサイズ
* 複数枚アップロード
* アップロード失敗時の再試行
* 他ユーザーの画像を参照できないこと
* Markdownとの関連付け

# 9. 復習カード機能
## 9.1 目的
学習セッションの文字起こしとAI分析から、復習用カードを生成する。
カードは学習セッションをまたいで利用し、同じ知識についてカードが無制限に増えないようにする。
## 9.2 カードの種類
初期版では最低限、以下を扱う。
* definition：用語の定義
* difference：違い
* condition：条件
* procedure：手順
* cause_effect：原因と結果
* component：構成要素
* example：具体例
* misconception：誤解しやすい点
* comparison：比較
## 9.3 カード例
```text
質問：
NAT Gatewayとは？
正解：
プライベートサブネット内のリソースが、外部からの着信を受けずにインターネットへ外向き通信するためのマネージドサービス。
補足：
- パブリックサブネットに配置する
- Elastic IPを関連付ける
```
## 9.4 生成枚数
カードを大量に生成しない。
目安として、1セッションにつき3〜10枚以内とする。
ただし、復習価値があるカードが3枚未満の場合、無理に3枚作らない。
優先対象：
* AIが曖昧と判定した内容
* 誤解の可能性がある内容
* 利用者が分からなかったと登録した内容
* 資格試験で重要な条件
* 他の知識の前提になる用語
* 利用者が明示的にカード化を希望した内容

# 10. 同じ「〇〇とは？」カードを1つにする機能
## 10.1 問題
AIがセッションごとにカードを生成すると、以下のような重複が発生する。
```text
NAT Gatewayとは？
NATゲートウェイとは何ですか？
NAT Gatewayの役割を説明してください。
NAT Gatewayとはどのようなサービスですか？
```
これらは表現が違っても、同じ定義を尋ねている。
新しいカードを毎回追加せず、1つの定義カードへ統合する。
## 10.2 conceptKey
質問文とは別に、概念を識別する`conceptKey`を保存する。
例：
```text
aws:nat-gateway
aws:internet-gateway
aws:cloudwatch
aws:cloudtrail
linux:inode
linux:symbolic-link
sr-labor:standard-monthly-remuneration
```
同じ`conceptKey`かつ同じ`cardType`の場合、原則として同一カード候補とする。
例：
```text
conceptKey: aws:nat-gateway
cardType: definition
```
以下は統合対象。
```text
NAT Gatewayとは？
NAT Gatewayの役割は？
NAT Gatewayを説明してください。
```
## 10.3 別カードにすべき例
同じ用語を扱っていても、問う知識が異なる場合は別カードとする。
```text
NAT Gatewayとは？
NAT Gatewayはどのサブネットに配置するか？
NAT GatewayとInternet Gatewayの違いは？
```
conceptKeyまたはcardTypeの例：
```text
aws:nat-gateway / definition
aws:nat-gateway-placement / condition
aws:nat-gateway-vs-internet-gateway / comparison
```
## 10.4 重複判定
以下を組み合わせて判定する。
1. `conceptKey`
2. `cardType`
3. 正規化した質問文
4. AIによる意味的類似判定
5. 既存カードの回答内容
質問文の完全一致だけで判定しない。
ただし、AIの類似判定だけで既存カードを自動削除しない。
## 10.5 重複候補画面
新規カード生成時、既存カードと似ている場合は次を表示する。
```text
既存カードと内容が近い可能性があります。
既存：
NAT Gatewayとは？
新規候補：
NAT Gatewayの役割を説明してください。
選択：
- 既存カードを更新
- 新規カードとして保存
- 内容を確認して統合
- 候補を破棄
```

# 11. 正解を1つに統合する機能
## 11.1 canonicalAnswer
同じ定義カードには、現在採用する正解を1つだけ持たせる。
フィールド名：
```text
canonicalAnswer
```
AIがセッションごとに異なる表現の答えを生成しても、復習時に複数の正解文を並べない。
## 11.2 正解と補足を分ける
`canonicalAnswer`は短く、単独で意味が通じる内容にする。
詳細条件は`supplement`へ分離する。
悪い例：
```text
NAT Gatewayとは、プライベートサブネットからインターネットへ接続するために使い、パブリックサブネットに置き、Elastic IPが必要で、AZごとに作ることが推奨され、料金も発生するサービス。
```
良い例：
```text
canonicalAnswer:
プライベートサブネット内のリソースが、外部からの着信を受けずにインターネットへ外向き通信するためのマネージドサービス。
```
```text
supplement:
- パブリックサブネットに配置する
- Elastic IPを関連付ける
- 高可用性を考慮する場合はAZごとに配置する
```
## 11.3 既存回答との統合
既存カードがある場合は次を行う。
1. 既存カードを取得
2. 新規回答候補と比較
3. 内容の重複を除く
4. 矛盾の有無を確認
5. 矛盾がなければ、より簡潔で正確な正解へ統合
6. 詳細は補足へ移す
7. 元セッションIDを追加
8. 更新履歴を保存
## 11.4 矛盾がある場合
既存回答と新規回答が明確に矛盾する場合、AIだけで黙って確定しない。
例：
```text
既存：
NAT Gatewayはプライベートサブネットに配置する。
新規：
NAT Gatewayはパブリックサブネットに配置する。
```
この場合：
```text
status: needs_review
```
とし、利用者に確認を求める。
表示例：
```text
回答候補に矛盾があります。
既存の正解：
NAT Gatewayはプライベートサブネットに配置する。
新しい候補：
NAT Gatewayはパブリックサブネットに配置する。
現在の正解として採用する内容を確認してください。
```
学習セッションのMarkdownには、誤った説明も当時の理解状態として残す。
カードの正解だけを修正可能とする。

```text

```
初回作成カードは、原則として翌日を初回復習日とする。

7. 次のカードへ進む

# 17. 前回との比較
同じ概念またはテーマについて過去の学習記録がある場合、AI分析で前回との変化を表示する。
例：
```text

今回：
ステートレスであることと、戻り通信の許可が必要なことを説明できた。
```
比較対象がない場合、無理に生成しない。
比較結果は、現在のセッションMarkdownに保存する。
過去のMarkdownは変更しない。

```text

未達成
判定不能
```
判定理由も短く表示する。

# 19. 認証・権限

* 他ユーザーのカードを取得・更新できない
* 他ユーザーのノート写真を参照できない

# 20. エラー処理
以下の失敗を区別して扱う。
* 入力エラー
* 認証エラー
* AI分析失敗
* Markdown生成失敗
* S3保存失敗
* DynamoDB登録失敗
* カード生成失敗
* 画像アップロード失敗
* 重複判定失敗
* ネットワークエラー
AI分析またはカード生成に失敗しても、元の文字起こしと学習情報を保存できる構成を優先する。
再実行できる処理には、再試行ボタンまたは再処理状態を用意する。
同じ操作を再実行した際、セッションやカードが重複作成されないよう、冪等性を考慮する。

利用者がスマートフォンまたはPCから、次の一連の操作を完了できる。
1. ログインする
2. 新しい学習セッションを開く

15. 次回復習日が更新される

* セッション詳細から内容を閲覧できる

## 23.3 カレンダー

* 学習した日に情報が表示される
* 日付からセッション一覧を開ける

* 学習時間とセッション数が正しく集計される

## 23.4 カード

## 28.1 既存システムとの関係【決定：b 土台に発展】

本プロジェクトは、既存の「Eureka Study」を土台に発展させる。

既存AWSリソース（すべて ap-northeast-1）：
- Lambda: `EurekaStudyApi`（Node.js、単一関数・action分岐方式）
- API Gateway: `https://xt65dgklb3.execute-api.ap-northeast-1.amazonaws.com/api`
- DynamoDB: `StudyLogTable`（PK: `USER#{userId}` / SK: `SESSION#...`, `JOB#...`, `WEAK#...`）
- S3: `study-log-web`（HTML配信元）, `eureka-study-attachments-naohiro`（写真、attachments/ 以下）
- CloudFront: `dkg9wfajfaxbc.cloudfront.net`（study-log-web を配信）
- Secrets Manager: `eureka-study/openai`（OPENAI_API_KEY）

既存コードは `legacy/` フォルダに置いた3ファイルを指す：
- `legacy/index.mjs` — Lambda本体（AI分析ジョブ方式、S3署名付きURL、JST日付処理、
  写真Vision読み取り、markCardUnderstood 等を含む最新版）
- `legacy/input.html` — 入力・AIログ作成ページ
- `legacy/summary.html` — まとめ・単元別・復習カードページ（重複統合 mergeCards 実装済み）

方針：
- 再利用候補：署名付きURLアップロード、toJstDate、OpenAIジョブ方式(generateLog/processGenerateLogJob/getGenerateLogJob)、
  understood フラグ、mergeCards の正規化ロジック（conceptKey設計の参考）
- 既存の学習データ（StudyLogTable 内の SESSION/JOB/WEAK アイテム）は削除しない
- 既存データのReRIKAI新形式への移行は本仕様の範囲外（別タスク）とする。
  ただし新形式の設計時に、旧SESSIONアイテムから将来移行可能であることを考慮する

## 28.2 技術スタック【決定：b 現行同様の素のHTML＋単一Lambda】

- フロントエンド：素のHTML/CSS/JavaScript（ビルド工程なし）。S3 + CloudFront で配信
- ページ分割は画面ごとに1HTMLファイルとする（現行の input.html / summary.html 方式を踏襲）
- バックエンド：Node.js 単一Lambda + API Gateway（HTTP API）、action分岐方式を踏襲
- ただし実装中に、この構成では仕様の実現が著しく困難と判断した場合は、
  代替案（例：ページ共通処理のJS分離、Lambda分割）を提案し、承認を得てから変更する（勝手に置き換えない）
- AI：OpenAI API。モデルは環境変数 OPENAI_MODEL（既定 gpt-4.1）。
  APIキーは既存 Secrets Manager `eureka-study/openai` を再利用する
- リージョン：ap-northeast-1
- §23.9 の「TypeScriptを使用している場合」は本構成では適用外。代わりに
  `node --check` による構文チェックと主要ロジックのNodeスクリプトによるテストを品質確認とする

## 28.3 認証【決定：a シングルユーザー簡易認証】

- 初期版はシングルユーザーとし、Cognitoは導入しない
- ただしデータ構造は userId 前提で設計する：
  - S3キー：`sessions/users/{userId}/...`, `notes/users/{userId}/...`
  - DynamoDB PK：`USER#{userId}`（現行と同じ形式。既定 userId は既存に合わせ `naohiro`）
  - API側で userId を固定値として付与し、クライアントから送られた userId は信用しない
- 画面アクセスの保護は現行運用（CloudFront配信＋非公開URL）を維持しつつ、
  可能であれば CloudFront Functions 等による簡易認証を追加してよい（新規有料サービスは不可）
- §23.1「ログインする」および§23.8の他ユーザー関連条件は、
  シングルユーザー構成では「簡易認証を通過する」「userId境界がキー設計に存在する」ことをもって満たすとする

## 28.4 インフラ管理【決定：a Terraform】

- 新規AWSリソースは Terraform で管理する（リポジトリ内 `terraform/` フォルダ）
- Claude Code は `terraform init` / `validate` / `plan` まで実行してよい。
  `terraform apply` は利用者が実行する
- 既存リソース（28.1のEureka Study分）はTerraform管理下に取り込まない（import しない。誤削除リスク回避）
- 既存リソースへの設定変更が必要な場合（例：S3バケットへのCORS追加）は、
  変更内容を提示し、利用者がコンソールまたはCLIで実施する

## 28.5 事前承認済みリソース

以下は §0 の「有料リソースの新規作成」の事前確認を不要とする
（実際の作成は terraform apply により利用者自身が行うため）：

- S3バケット：3個以内
- DynamoDBテーブル：オンデマンド課金モードのもの
- Lambda関数
- API Gateway（HTTP API）
- CloudFront ディストリビューション

これ以外の有料リソース（Cognito、OpenSearch、RDS、Bedrock、ElastiCache、NAT Gateway等）は事前確認を要する。

## 28.6 重複判定の初期版範囲

§10.4「AIによる意味的類似判定」は、初期版では埋め込みベクトル検索を導入しない。
重複判定は以下で実現する：

1. conceptKey（カード生成時にAIが付与）
2. cardType
3. 正規化した質問文（小文字化・空白除去・句読点/疑問符除去。legacy/summary.html の
   normalizeCardText と同等のロジックを参考にする）
4. カード生成プロンプト内で、同一分野の既存カード一覧をAIに渡して照合させる方式は使用してよい

ベクトル検索基盤（OpenSearch等）の導入は将来対応とする。

## 28.7 understandingScoreの扱い

理解度スコアを保存・表示する場合は、必ず判定理由（§18）とセットで生成・表示する。
スコア単独の表示はしない。スコア生成が不安定な場合は、初期版ではスコア自体を省略してよい。

## 28.8 §0への補足

§0 の「まず既存コードベース、既存AWS構成…を調査してください」における既存コードベースとは、
本リポジトリの `legacy/` フォルダ（28.1参照）を指す。
AWS構成はAWSアカウントへの読み取りアクセスではなく、28.1の記載を一次情報として扱ってよい
（AWS CLIが利用可能な場合、read-onlyの確認コマンドは実行してよい）。
